#### P1-1: Soft-Auth Fallback — Fail-Open Risk

**Finding:** A "soft-auth fallback" exists in the auth flow. The critical question is whether this fails **closed** (denies access, returns 401) or **open** (grants limited/anonymous access) when no valid JWT is present.

**Fail-open behaviour in a medical study platform means:**
- Any request without a valid session token gets served content
- RLS policies that rely on `auth.uid()` return NULL — and any policy written as `auth.uid() = user_id` will correctly deny, but any policy using `OR auth.uid() IS NULL` or any table without RLS enabled will expose data

**Required audit steps (Engineering agent):**
1. Locate the soft-auth middleware (likely in `middleware.ts` or an auth helper in `lib/`). Document the exact fallback behaviour.
2. Run this query against Supabase to identify tables without RLS enabled:
```sql
SELECT tablename
FROM pg_tables
WHERE schemaname = 'public'
AND tablename NOT IN (
  SELECT DISTINCT tablename
  FROM pg_policies
  WHERE schemaname = 'public'
);
```
3. For every table with RLS enabled, run:
```sql
SELECT tablename, policyname, cmd, qual, with_check
FROM pg_policies
WHERE schemaname = 'public'
ORDER BY tablename;
```
Review every policy where `qual` contains `IS NULL` or `TRUE` — these may allow unauthenticated reads.
4. Test explicitly: make a Supabase client request with no auth token to each major table (`mcq_questions`, `saqs`, `viva_questions`, `document_chunks`, `study_notes`). Assert 401 or empty result — never data.

**Expected RLS policy shape for all user-data tables:**
```sql
-- Correct: fails closed for unauthenticated requests
CREATE POLICY "user_owns_row" ON user_sessions
  FOR ALL USING (auth.uid() = user_id);
-- If auth.uid() is NULL (no session), this evaluates to NULL = user_id → false → denied
```

## RISK 1: COPYRIGHT & IP — EXISTENTIAL 🔴

### The problem
57 books digitised, chunked into 333,804 pgvector rows, and served as product output. ANZCA/ACEM/CICM past exam questions replicated, AI-enriched, and distributed. No licensing evidence on file.

### Australian Copyright Act 1968 exposure
- **Digitising and embedding** a copyrighted text without licence = reproduction (s.31). Fair dealing exceptions (s.40 research/study) are *personal*, non-commercial, and limited in quantity — they do not cover systematic bulk digitisation for a commercial SaaS platform.
- **RAG serving of excerpts** in response to queries = communication to the public (s.31(1)(a)(iv)). Each query that returns a book passage is a potential infringement event.
- **AI-enriched exam questions** derived from college past papers: the underlying questions are likely owned by ANZCA/ACEM/CICM. Enriching them with Claude does not launder the IP — derivative works require consent of the original copyright holder.
- **The 333,804 chunks are not a defence — they are 333,804 instances of exposure.**

### College IP specifically
ANZCA, ACEM, and CICM own their syllabi, learning objectives, and assessment bank as institutional IP. "ANZCA 330/330 DONE" means their full published curriculum has been ingested. None of these colleges have been contacted. A cease-and-desist from any one of them removes the content foundation the entire product rests on.

### Immediate actions required
1. **Legal counsel must audit every source title** against licensing status before any further enrichment spend ($465 ANZCA + $270-410 CICM remaining = up to $875 at risk).
2. **Pause enrichment pipeline** pending copyright clearance opinion — spending $875 more on infringing content compounds liability.
3. **Engage ANZCA, ACEM, CICM directly** — a partnership/licensing conversation is far better than a cease-and-desist. Rafael as an ANZCA candidate has a natural entry point. Frame as: *"We want to be your official prep platform."*
4. **Fair use / fair dealing opinion** — instruct IP counsel to assess whether any of the 57 books fall under educational statutory licences (Part VB, Copyright Act) and whether a CAL (Copyright Agency Limited) licence covers this use case.

### CEO instruction to Engineering
Do not run further enrichment pipelines until Legal clears the source IP status. This is a hard stop.

## RISK 3: TGA SOFTWARE AS A MEDICAL DEVICE — CONTINGENT 🟡

### The problem
TGA's SaMD framework (aligned to IMDRF) classifies software as a medical device if it meets the *intended purpose* test — i.e., if it is intended to be used for diagnosis, prevention, monitoring, prediction, prognosis, treatment, or alleviation of disease/injury.

### Current technical description: LOW risk
"AI-powered exam preparation platform" — this framing is *educational*, not clinical. Exam prep does not meet the intended purpose test on its face.

### What triggers TGA classification
The trigger is **marketing language**, not product function. If any of the following appear on the landing page, LinkedIn, medical forums, or app copy:
- "Become a safer anaesthetist"
- "Improve clinical decision-making"
- "Reduce errors in ICU"
- "Evidence-based preparation linked to clinical outcomes"

...then the *intended purpose* shifts from education to clinical benefit, and TGA classification risk activates. Class I SaMD (low risk) still requires ARTG inclusion and conformity assessment.

### The VIVA mode is the highest-risk feature
A voice-based "AI examiner" that role-plays clinical scenarios (e.g., managing a difficult airway, treating anaphylaxis) and provides feedback on trainee responses is closer to a clinical simulation tool than a flashcard app. If any VIVA question involves active patient management scenarios and the feedback system claims to assess clinical competence, TGA scrutiny increases.

### Immediate actions required
1. **Legal counsel must review all current live copy** — landing page, any social posts, any forum comments Rafael has made — for clinical outcome framing. Provide screenshots.
2. **Establish a written intended purpose statement** ("Synapse is an educational platform for exam preparation. It does not assess clinical competence or fitness to practise.") and embed this in ToS and landing page.
3. **Do not add any feature that outputs a clinical recommendation or patient management plan** without separate TGA assessment.
4. If Rafael has posted in any ANZCA trainee forums linking exam performance to clinical outcomes, those posts must be reviewed.
